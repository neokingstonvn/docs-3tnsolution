🎯 Mục tiêu Kế hoạch MVP (Ứng dụng Task Kingston): Xây dựng nền tảng quản lý công việc tối giản phục vụ văn phòng công ty với 3 phân quyền chính (Admin, Project Manager, Staff). Hệ thống tập trung giải quyết các bài toán: Phân quyền tạo dự án, giao việc trực quan qua Kanban Board, khóa trạng thái Done thông qua luồng phê duyệt nhiệm vụ nghiêm ngặt, tự động ghi nhận nhật ký hoạt động (Activity Log) và thảo luận trực tiếp (Comments). Phiên bản MVP chạy hoàn toàn độc lập với database mock ở client-side (`localStorage`) để duyệt UAT siêu tốc trước khi kết nối dữ liệu thực tế.

1. Tại Sao Xây Dựng Hệ Thống Chuyên Biệt?

Tiêu Chí Đánh Giá Lợi Ích Và Ý Nghĩa Thực Tế
Khóa trạng thái Done Nhân viên (Staff) không thể tự ý đánh dấu hoàn thành (Done) mà chỉ có thể gửi yêu cầu duyệt. Công việc chỉ được chuyển sang Done khi PM/Admin bấm nút duyệt thực tế.
Lưu vết Activity Log Tự động ghi lại mọi thay đổi quan trọng của công việc (người gán, cập nhật trạng thái, ngày gửi yêu cầu duyệt) giúp minh bạch hóa trách nhiệm.
Phân quyền RBAC sâu Tách biệt hoàn toàn tính năng theo vai trò: Admin tạo dự án và phân quyền; PM tạo task và duyệt báo cáo; Staff nhận việc và nộp báo cáo kết quả.
Độ đo tiến độ tự động Phần trăm hoàn thành dự án được tính toán tự động dựa trên tỷ lệ số lượng task đã chuyển sang trạng thái Done thực tế, tránh các báo cáo ảo.

4. Thiết Kế Cơ Sở Dữ Liệu Rút Gọn

Để đáp ứng nhu cầu cốt lõi, cơ sở dữ liệu của Task Kingston gồm các bảng tinh gọn sau:

1. Table `users` (Nhân sự & Vai trò)
- id: UUID PRIMARY KEY (Định danh duy nhất)
- email: VARCHAR(100) UNIQUE (Email đăng nhập)
- full_name: VARCHAR(100) (Tên đầy đủ)
- role: VARCHAR(20) CHECK (role IN ('Admin', 'Project Manager', 'Staff'))

2. Table `projects` (Dự án)
- id: UUID PRIMARY KEY
- name: VARCHAR(150) NOT NULL (Tên dự án)
- description: TEXT
- pm_id: UUID REFERENCES users(id) (PM phụ trách)
- status: VARCHAR(20) CHECK (status IN ('Active', 'On Hold', 'Completed'))

3. Table `tasks` (Công việc)
- id: UUID PRIMARY KEY
- project_id: UUID REFERENCES projects(id) ON DELETE CASCADE
- title: VARCHAR(255) NOT NULL
- assignee_id: UUID REFERENCES users(id) ON DELETE SET NULL
- priority: VARCHAR(20) CHECK (priority IN ('High', 'Medium', 'Low'))
- due_date: TIMESTAMP
- status: VARCHAR(20) CHECK (status IN ('To Do', 'In Progress', 'Reviewing', 'Done'))

4. Table `approval_requests` (Hàng đợi Phê Duyệt Trạng Thái)
- id: UUID PRIMARY KEY
- task_id: UUID REFERENCES tasks(id) ON DELETE CASCADE
- submitter_id: UUID REFERENCES users(id)
- report_text: TEXT (Nội dung báo cáo công việc)
- report_link: VARCHAR(255) (Link tài liệu báo cáo)
- status: VARCHAR(20) CHECK (status IN ('Pending', 'Approved', 'Rejected'))
- feedback: TEXT (Ý kiến phản hồi từ PM/Admin khi Reject)

5. Table `activity_logs` (Nhật ký hoạt động & Bình luận)
- id: UUID PRIMARY KEY
- task_id: UUID REFERENCES tasks(id) ON DELETE CASCADE
- user_id: UUID REFERENCES users(id)
- action: TEXT (Hành động hoặc nội dung bình luận)
- created_at: TIMESTAMP DEFAULT NOW()

5. Quy Trình Vận Hành Thao Tác

Luồng A: Khởi tạo và Phân quyền Dự án

Admin khởi tạo dự án mới trên Dashboard ➔ Chỉ định Project Manager phụ trách ➔ PM cấu hình các thành viên Staff tham gia dự án ➔ Thành viên tham gia bắt đầu thấy dự án xuất hiện trong danh sách cá nhân.

Luồng B: Giao việc & Kéo thả trạng thái

PM tạo công việc (Task) trong màn hình chi tiết dự án ➔ Thiết lập mức độ ưu tiên và ngày hạn chót ➔ Gán cho Staff ➔ Staff chuyển trạng thái task từ To Do sang In Progress trên Kanban Board để báo hiệu bắt đầu làm việc.

Luồng C: Đệ trình & Phê duyệt Hoàn thành

Khi làm xong, Staff điền Link báo cáo và nội dung giải trình ➔ Nhấn gửi duyệt để đưa trạng thái sang Reviewing (Khóa quyền sửa/kéo thả của Staff) ➔ PM/Admin nhận thông báo trong Hàng đợi, bấm Approve (Task chuyển sang Done và tự động tính lại % tiến độ dự án) hoặc Reject (Task trả về In Progress kèm phản hồi lý do).

6. Kiến Trúc Phát Triển Custom Code

Mã nguồn dự án được thiết kế phân lớp khoa học để tối ưu tính mở rộng:

1. Giao diện Người dùng (Frontend Layer)
- HTML5 thuần phân tách hoàn toàn mã Javascript và CSS.
- CSS variables quản lý hệ thống màu sắc sáng Slate v2 thanh lịch thống nhất.
- Router xử lý điều hướng mượt mà, tải trang nhanh không gây lag.

2. Lớp Dữ Liệu Giả Lập (Mock DB Layer - db.js)
- Đọc ghi dữ liệu đồng bộ thông qua APIs `localStorage`.
- Đảm bảo tính nhất quán dữ liệu qua lại giữa các vai trò (khi Admin đổi tên dự án hoặc Staff nộp báo cáo, PM lập tức cập nhật dữ liệu tương ứng).
- Sẵn sàng chuyển đổi sang các REST APIs kết nối CSDL thực tế.

7. Lộ Trình Triển Khai MVP (Roadmap)

Phase 1: Setup Nền Tảng & Cơ Sở Dữ Liệu

Thiết lập Clean Architecture, tạo database DDL SQL và cấu hình mock dữ liệu mẫu (seed data) trong file db.js để khởi chạy lần đầu.

Phase 2: APIs & Kanban Board Giao Diện

Hiện thực các luồng CRUD Dự án và Công việc. Thiết kế Kanban Board kéo thả hỗ trợ chuyển trạng thái trực quan.

Phase 3: Log Engine & Bình Luận Trao Đổi

Phát triển log engine tự động lưu vết các thao tác đổi trạng thái hoặc người phụ trách. Thêm khung bình luận trao đổi thông tin trong task.

Phase 4: Luồng Phê Duyệt Nhiệm Vụ

Hoàn thiện Hàng đợi phê duyệt (Approval Queue). Khóa quyền tự ý đổi trạng thái Done của Staff và thêm form feedback từ chối của PM.

Phase 5: Kiểm Thử UAT & Bàn Giao

Chạy kịch bản kiểm thử Happy Paths và hoàn thành các cổng Portal UAT tương tác hoàn chỉnh.
